# GUIA: Registro de Ideias e Documentação

## Quando usar
- Quando o usuário propõe novas funcionalidades ou visões para o futuro do projeto.
- Quando é necessário atualizar a base de conhecimento (knowledge base) sem alterar o código funcional.

## Checklist obrigatório
- [ ] Ler a proposta do usuário completamente.
- [ ] Verificar se o arquivo `docs/knowledge/IDEIAS_DE_IMPLANTACAO.md` existe.
- [ ] Criar ou atualizar `docs/execution/current-task.md` antes de qualquer edição.

## Execução padrão
1. Criar/Atualizar `current-task.md`.
2. Ler `IDEIAS_DE_IMPLANTACAO.md` para entender a estrutura atual.
3. Adicionar as novas ideias seguindo o padrão de formatação (Markdown, seções, bullet points).
4. Usar linguagem que preserve a "vibe" e o entusiasmo do usuário (Matadora, Killer Idea, etc).
5. Confirmar a gravação com o usuário.
6. Mover `current-task.md` para `/docs/history/YYYY-MM-DD-nome-da-tarefa.md`.

## Erros comuns
- Esquecer de criar o `current-task.md`.
- Alterar o código em vez de apenas documentar a ideia (a menos que solicitado).
- Perder detalhes importantes da fala do usuário.

## Estratégias de fallback
- Se o arquivo de ideias estiver muito grande, sugerir a criação de subarquivos por categoria.

## Restrições
- Não deletar ideias antigas sem autorização explícita.
- Não implementar a funcionalidade imediatamente, apenas documentar.
